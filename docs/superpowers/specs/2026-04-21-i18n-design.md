# Internationalization (i18n) — Design Spec

## Context

AIQuiz is a certification exam prep platform preparing for public launch. All user-facing text is currently hardcoded in English across ~25 files. Adding internationalization enables Portuguese (pt) and English (en) support, broadening the target audience for launch.

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Persistence | localStorage | Simple, no URL restructuring, no server-side changes |
| Scope | UI text only | Certification/topic names are user-generated, stay as-is |
| File format | `.properties` (key=value) | User preference; one file per language |
| Architecture | Context + Reducer + custom parser | Matches existing project patterns, no external dependencies |
| Default language | `pt` | Primary target audience |

### Data Flow

```
App mounts
  → LanguageProvider reads localStorage("app-language") (default: "pt")
  → fetch("/messages/{language}.properties") to load raw text (files served from public/ or imported via raw loader)
  → properties-parser converts to Record<string, string>
  → Dispatch SET_MESSAGES → state.messages updated
  → Components call t("key") → lookup in state.messages
  → Returns translated string or key itself as fallback

User clicks LanguageSwitch
  → Dispatch SET_LANGUAGE("en")
  → Save to localStorage
  → Re-import and parse new .properties file
  → Dispatch SET_MESSAGES → entire tree re-renders with new strings
```

### Properties File Format

```properties
# Namespace: page.section.element
homepage.hero.title=AI-Powered Exam Prep
homepage.hero.subtitle.line1=Stop memorizing.

# Interpolation with {variable}
quiz.progress.value={answered} of {total}
toast.savedSuccessfully="{title}" saved successfully.
error.duplicate=Code "{code}" already exists.
```

### Key Namespace Convention

| Prefix | Scope |
|---|---|
| `common.*` | Shared UI (buttons, nav, footer) |
| `homepage.*` | Homepage sections |
| `login.*` | Login page |
| `certification.*` | Configure Certification page |
| `generate.*` | Generate Questions page |
| `quiz.*` | Quiz page |
| `error.*` | Validation and error messages |
| `toast.*` | Toast notification titles |
| `aria.*` | Accessibility labels |

### Properties Parser

Handles:
- `key=value` pairs (one per line)
- Lines starting with `#` are comments
- Empty lines are ignored
- Values can contain `=` (only first `=` is the delimiter)
- Trims whitespace from keys and values

### useTranslation Hook API

```typescript
const { t, language, setLanguage } = useTranslation();

t("common.signIn")                           // → "Sign In"
t("quiz.progress.value", { answered: 5, total: 20 }) // → "5 of 20"
t("nonexistent.key")                         // → "nonexistent.key" (fallback)
```

### LanguageSwitch Component

- Toggle button (not dropdown — only 2 languages)
- Shows flag emoji + language code: 🇧🇷 PT / 🇺🇸 EN
- Positioned between ThemeSwitch and auth section in navbar
- Same styling approach as ThemeSwitch
- Present in both desktop and mobile navbar layouts

### config/site.ts Handling

Navigation labels in `siteConfig.navItems` are currently hardcoded strings. Two options:

**Chosen approach:** Store translation keys in `siteConfig` (e.g., `label: "nav.generateQuestions"`) and resolve them in the navbar via `t(item.label)`. This keeps `siteConfig` as the single source for routes while delegating display text to `.properties`.
